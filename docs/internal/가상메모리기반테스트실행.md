# 가상 메모리 기반 테스트 실행 (`feature-virtual-memory`)

> **대상 버전**: 0.5.0 예정  
> **관련 브랜치**: `feature-virtual-memory`  
> **핵심 API**: `module.registerHooks()` (Node.js v22.15+)

---

## 1. 개요: 왜 바꿔야 했는가?

### 기존 방식의 문제

`@dannysir/js-te`는 모킹(mocking) 기능을 구현하기 위해 Babel을 이용해 사용자의 소스 코드를 변환한다.  
변환된 코드는 `import`/`require`를 **wrapper 패턴**으로 교체하여, 실행 시점에 `mockStore`를 조회하도록 만든다.

기존 방식은 이 변환 결과를 **사용자의 원본 파일에 직접 덮어쓴** 뒤, 테스트가 끝나면 `restoreFiles()`로 복구하는 방식이었다.

```
[ 기존 흐름 ]

setupFiles()
  → fs.writeFileSync(game.js, 변환된코드)      ← 디스크 오염 ⚠️
  → fs.writeFileSync(random.js, 변환된코드)    ← 디스크 오염 ⚠️

runTests()
  → fs.writeFileSync(basic.test.js, 변환된코드) ← 디스크 오염 ⚠️
  → await import('./basic.test.js')

finally
  → fs.writeFileSync(game.js, 원본코드)        ← 복구
  → fs.writeFileSync(random.js, 원본코드)      ← 복구
  → fs.writeFileSync(basic.test.js, 원본코드)  ← 복구
```

이 방식에는 세 가지 치명적인 문제가 있었다.

| 문제 | 설명 |
|---|---|
| **비정상 종료 시 소스 훼손** | `SIGKILL`, OOM 등으로 프로세스가 죽으면 `finally`가 실행되지 않아 변환된 코드가 원본 파일에 남는다 |
| **읽기 전용 파일 대응 불가** | `chmod 444`로 보호된 파일이 있으면 `EACCES` 오류로 테스트 자체가 실패한다 |
| **불필요한 파일 eager 변환** | `setupFiles()`가 mock 여부와 관계없이 **모든 소스 파일**을 미리 변환해 디스크에 쓴다 |

### 개선 목표

> 사용자의 원본 파일을 **0바이트도 건드리지 않고**, 메모리에서만 변환된 코드를 Node.js에 공급한다.

## 2. 핵심 원리: `module.registerHooks()`

### Node.js 모듈 로딩 파이프라인

Node.js는 파일을 불러올 때 내부적으로 다음 두 단계를 거친다.

```
resolve(specifier) → load(url) → 실행
```

- **resolve**: `import './game.js'` 같은 specifier를 절대 URL로 변환
- **load**: URL에 해당하는 파일을 디스크에서 읽어 소스 코드를 반환
- **실행**: 반환된 소스를 컴파일하고 평가(evaluate)

`module.registerHooks()`는 이 파이프라인의 **`load` 단계**를 가로채서 커스텀 소스를 반환할 수 있게 해주는 API다.

```js
import { registerHooks } from 'node:module';

registerHooks({
  load(url, context, nextLoad) {
    const result = nextLoad(url, context); // Node가 디스크에서 읽은 원본 소스
    const transformed = myTransform(result.source);
    return { ...result, source: transformed }; // 변환된 소스를 메모리에서 반환
  }
});
```

파일 시스템에는 **읽기(read)만** 발생하고, **쓰기(write)는 발생하지 않는다.**

### `module.register()` 와의 차이

기존의 `module.register()` 방식은 worker 스레드에서 로더를 실행하는 반면,  
`module.registerHooks()`는 **메인 스레드에서 동기적으로** 실행된다.

| 항목 | `registerHooks()` ✅ | `register()` |
|---|---|---|
| 실행 스레드 | 메인 스레드 | 별도 worker 스레드 |
| 동기/비동기 | 동기 | 비동기 |
| Closure 접근 | 가능 | 불가 (구조복제만) |
| 커버 범위 | **ESM + CJS 단일 훅** | ESM만 |
| 해제 | `deregister()` 반환 | 불가 |

메인 스레드에서 실행되기 때문에 `mockedPaths`를 **closure로 직접 캡처**할 수 있어,  
worker 스레드에 데이터를 넘기기 위한 복잡한 `MessageChannel` 처리가 불필요하다.

또한 이 훅은 ESM `import`, 동적 `import()`, **CJS `require()`** 전부를 가로채므로,  
기존에 별도로 필요했던 `Module.prototype._compile` 몽키 패치도 필요 없다.

## 3. 변경된 전체 흐름

### 3.1 개요 다이어그램

```
사용자가 js-te 실행

bin/cli.js
  │
  ├─ (1) setupEnvironment()
  │       └─ 사용자 프로젝트 타입(ESM/CJS) 감지
  │       └─ @dannysir/js-te 로드 → global에 test/describe/expect/mock 등 주입
  │
  ├─ (2) setupFiles()
  │       └─ findTestFiles()     → *.test.js, test/ 폴더 탐색
  │       └─ collectMockedPaths() → 테스트 파일을 AST 파싱해 mock() 경로만 수집
  │       ※ 기존: 전체 소스 eager 변환 → 완전 제거
  │
  ├─ (3) installLoaderHook(mockedPaths)   ← 신규 ★
  │       └─ registerHooks({ load }) 등록
  │       └─ 이후 모든 import/require가 훅을 거침
  │
  ├─ (4) runTests()
  │       └─ 파일마다 반복:
  │           ├─ await import(testFile)  ← 훅 발동
  │           └─ jsTe.run()
  │
  └─ process.exit(exitCode)
     ※ 기존 finally { restoreFiles() } 제거
```

### 3.2 단계별 상세 설명

#### 1단계: 환경 구성 (`setupEnvironment`)

사용자 프로젝트의 `package.json`에서 `"type"` 필드를 읽어  
ESM 프로젝트인지 CJS 프로젝트인지 판별한다.

```js
// src/cli/setupEnvironment.js
const getUserModuleType = () => {
  const pkg = JSON.parse(fs.readFileSync('package.json', 'utf-8'));
  return pkg.type === 'module' ? MODULE_TYPE.ESM : MODULE_TYPE.CJS;
};
```

판별 결과에 따라 `@dannysir/js-te`를 ESM `import` 또는 CJS `require`로 불러온 뒤,  
`test`, `describe`, `expect`, `mock`, `fn`, `beforeEach` 등을 `global`에 주입한다.  
덕분에 사용자 테스트 파일은 별도 import 없이 이 API들을 바로 사용할 수 있다.

#### 2단계: 파일 탐색 & mock 경로 사전 수집 (`setupFiles`)

```js
// src/cli/setupFiles.js (변경 후)
export const setupFiles = () => {
  const testFiles = findTestFiles(process.cwd());
  const mockedPaths = collectMockedPaths(testFiles);  // AST 파싱만 (실행 X)
  return { mockedPaths, testFiles };
};
```

`collectMockedPaths()`는 각 테스트 파일을 **Babel로 파싱만** 하여  
`mock('경로', ...)` 호출의 첫 번째 인자를 절대경로로 변환해 `Set<string>`에 담는다.

이 정보는 이후 로더 훅이 **"이 파일을 변환할 것인가?"** 를 판단하는 데 쓰인다.  
mock이 걸리지 않는 파일은 Babel 변환 자체를 스킵하므로 불필요한 비용이 없다.

> **기존과의 차이**  
> 기존 `setupFiles()`는 `findAllSourceFiles()`로 모든 소스 파일을 찾아  
> `transformFiles()` → `fs.writeFileSync()`로 전부 디스크에 덮어썼다.  
> 이 로직이 **완전히 제거**되었다.

#### 3단계: 메모리 훅 설치 (`installLoaderHook`) ★

```js
// src/cli/loaderHook.js
export const installLoaderHook = (mockedPaths) => {
  return registerHooks({
    load(url, context, nextLoad) {
      const result = nextLoad(url, context);       // Node가 디스크 read
      if (!shouldTransform(url, mockedPaths)) return result;

      const source = extractSource(result);
      const filename = fileURLToPath(url);
      const transformed = transformSource(source, filename, mockedPaths);

      return { ...result, source: transformed, shortCircuit: true };
    },
  });
};
```

`mockedPaths`는 **closure로 캡처**되어 훅 내부에서 바로 접근한다.  
훅이 등록된 이후부터 `import`/`require`가 발생할 때마다:

1. Node가 디스크에서 원본 소스를 읽어 `result.source`로 전달
2. `shouldTransform()`으로 변환 필요 여부 판단
   - `node_modules` 경로 → skip
   - `.js` / `.mjs`가 아닌 파일 → skip
   - `mockedPaths`가 비어 있음 → skip (mock 없는 프로젝트는 변환 비용 0)
3. 변환이 필요한 경우 `transformSource()`로 Babel 적용
4. 변환된 소스를 메모리에서 Node에 반환 → **파일 시스템 write 없음**

**중요**: `resolve` 훅은 구현하지 않는다.  
경로 해석(`./game.js` → 절대경로)은 Node 기본 동작에 맡기고,  
우리는 **"소스 바꿔치기(`load`)"에만 관여**한다.

#### 4단계: 테스트 실행 루프 (`runTests`)

```js
// src/cli/runTests.js (변경 후)
for (const file of testFiles) {
  await import(pathToFileURL(path.resolve(file)).href);  // 훅 발동
  const { passed, failed } = await jsTe.run();
}
```

`await import(testFile)` 호출 시 Node 로더가 `load` 훅을 호출하며  
아래의 연쇄 변환이 **메모리 안에서만** 자동으로 발생한다:

```
import(basic.test.js)
  └─ load 훅 발동 → Babel 변환 (wrapper 패턴 적용)
       └─ import('./game.js') 처리 중
              └─ load 훅 발동 → Babel 변환
       └─ import('./random.js') 처리 중
              └─ load 훅 발동 → Babel 변환
```

테스트 파일 실행 중:
- `describe(...)`, `test(...)` 호출 → `testManager` 내부 배열에 등록
- `mock('/abs/path.js', {...})` 호출 → `mockStore` Map에 저장

`jsTe.run()` 호출 시:
- `testManager`에 등록된 테스트를 순차 실행
- wrapper 함수가 매 호출마다 `mockStore.has(path)` 체크
  - `true` → mock된 값 반환
  - `false` → 원본 모듈 함수 호출
- `expect(...)` 판정 → pass/fail 집계
- 각 테스트 종료 시 `clearAllMocks()` → `mockStore` 비움

#### 5단계: 결과 출력 & 종료

```js
// bin/cli.js (변경 후)
const main = async () => {
  try {
    const jsTe = await setupEnvironment();
    const { mockedPaths, testFiles } = setupFiles();

    installLoaderHook(mockedPaths);

    const { totalPassed, totalFailed } = await runTests(jsTe, mockedPaths, testFiles);
    // ...
  } catch (error) {
    console.log(getErrorMsgInLogic(error.message));
    return NUM.ONE;
  }
  // finally { restoreFiles() } ← 제거됨
};
```

`finally { restoreFiles() }` 블록이 완전히 사라졌다.  
디스크를 건드린 적이 없으므로 복구가 불필요하기 때문이다.

## 4. Babel 변환 상세: wrapper 패턴

로더 훅 안에서 `transformSource()`가 호출되며,  
내부적으로 기존 `babelTransform` 플러그인을 **그대로 재사용**한다.

### 변환 전

```js
import { random } from './random.js';
```

### 변환 후 (wrapper 패턴)

```js
const mockStore = global.mockStore;
const _original = await import('./random.js');
const random = (...args) => {
  const module = mockStore.has('/abs/path/random.js')
    ? { ..._original, ...mockStore.get('/abs/path/random.js') }
    : _original;
  return module.random(...args);
};
```

`random` 함수가 **호출될 때마다** `mockStore`를 확인하므로,  
`mock()` 등록 순서와 `import` 순서에 관계없이 올바른 값이 반환된다.

### `transformSource.js` 캐시

동일한 파일을 여러 테스트에서 사용할 경우 중복 변환 비용을 막기 위해  
`filename + 소스 길이 + 해시` 기반 캐시를 두었다.

```js
// src/cli/utils/transformSource.js
const cache = new Map();  // key: `${filename}:${length}:${hash}` → transformedCode

export const transformSource = (code, filename, mockedPaths) => {
  if (mockedPaths.size === 0) return code;  // mock 없으면 Babel 비용 0

  const cacheKey = `${filename}:${code.length}:${hashCode(code)}`;
  if (cache.has(cacheKey)) return cache.get(cacheKey);

  const { code: transformed } = transformSync(code, { ... });
  cache.set(cacheKey, transformed);
  return transformed;
};
```

## 6. 기존 방식 vs 신규 방식 비교

| 시점 | 기존 (디스크 덮어쓰기) | 신규 (메모리 훅) |
|---|---|---|
| `setupFiles` | 모든 소스 파일 `writeFileSync`로 덮어쓰기 | mock 경로만 수집, 디스크 무변경 |
| `runTests` 진입 직전 | — | `registerHooks({ load })` 설치 |
| 테스트 파일 로드 | 이미 변환된 디스크 코드를 `import` | `import` 순간 훅이 메모리에서 변환 공급 |
| 하위 의존 모듈 로드 | 이미 변환된 디스크 코드 | 훅이 재귀적으로 즉석 변환 |
| 종료 `finally` | `restoreFiles()`로 원본 복구 필요 | **아무 작업도 안 함** |
| 비정상 종료 시 | 변환 코드가 사용자 소스에 잔존 | **원본 파일 0바이트도 바뀐 적 없음** |
| 읽기 전용 파일 | `EACCES` 오류로 테스트 실패 | 읽기만 하므로 정상 동작 |

## 7. 제약 사항

- **Node.js v22.15.0 이상** 필요 (`module.registerHooks` 도입 버전)
- `package.json`에 `"engines": { "node": ">=22.15.0" }` 이 명시되어 있으므로 하위 버전에서는 경고가 발생한다
- `module.registerHooks`는 현재 **Release Candidate** 상태 (Node v25.4.0 기준)이며, 추후 stable로 전환될 예정
